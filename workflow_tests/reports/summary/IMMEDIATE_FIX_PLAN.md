# 立即修复清单

本文件只列“现在马上可以优化”的内容，目标是先把不可上线风险降下来。优先级按 P0、P1 排列。

## 优先级依据

本清单基于 336 条 UTF-8 有效测试结果生成。关键数据如下：

| 风险点 | 相关流程 | 测试表现 |
|---|---|---|
| 转人工失败 | HUMAN_HANDOFF | 24 条中 15 条出现欢迎语兜底，明确人工/投诉/退款/强议价大量未承接 |
| 标准短回复失败 | DIRECT_REPLY | 24 条中 6 条欢迎语兜底，到店前问题和短确认失败明显 |
| 门店事实编造 | SF6 | 24 条均接口成功，但多条编造地址、营业时间、停车、地铁、门店数量 |
| 活动事实编造 | SF8 | 24 条中 1 条执行异常、1 条空回复、4 条欢迎语兜底，并大量编造活动 |
| 价格缺失和内部词 | SF7 | 24 条无具体价格输出，2 条欢迎语兜底，存在系统价/系统配置类表达 |
| 预约确认风险 | SF9 | 24 条中 3 条欢迎语兜底，存在无接口结果直接确认/取消/定金承诺 |

因此，当前不是先优化语气风格，而是先修安全出口、外部数据边界和兜底节点。

## P0-1 修复 HUMAN_HANDOFF 闭环

### 问题

明确要人工、投诉、退款、维权、曝光、强议价、高风险人群、自伤风险、病例/伤口图片等大量输出欢迎语兜底。

### 立即修改

1. 在工作流中建立真正的 `HUMAN_HANDOFF` 输出节点。
2. 主路由输出以下任一字段时，后续不得进入任何业务子流程：
   - `need_human=true`
   - `subflow=HUMAN_HANDOFF`
   - `intent=human_request`
   - `intent=complaint_refund`
   - `intent=bargain` 且为强压价
3. `HUMAN_HANDOFF` 节点禁止使用普通欢迎语。

### 建议输出模板

明确要人工：

```json
{
  "reply_messages": [
    {
      "type": "text",
      "order": 1,
      "content": "可以，我帮你安排顾问接上。"
    }
  ]
}
```

投诉/退款：

```json
{
  "reply_messages": [
    {
      "type": "text",
      "order": 1,
      "content": "我理解你现在比较不满，这个我帮你转人工同事处理。"
    }
  ]
}
```

严重术后异常：

```json
{
  "reply_messages": [
    {
      "type": "text",
      "order": 1,
      "content": "这个情况需要尽快让护理老师或门店同事确认，我先帮你转人工。"
    }
  ]
}
```

高风险人群：

```json
{
  "reply_messages": [
    {
      "type": "text",
      "order": 1,
      "content": "你这个情况需要人工同事进一步确认，不建议直接线上判断。"
    }
  ]
}
```

## P0-2 强化主路由高风险规则

### 必须前置拦截

以下内容必须在主路由最前面判断，命中后不允许被历史对话覆盖：

- 投诉、退款、维权、曝光、效果纠纷
- 明确要求人工、真人、顾问、客服接待
- 孕期、哺乳期、未成年、严重慢病、严重过敏
- 自杀、自残、不想活
- 病例、体检报告、诊断证明、异常伤口图片
- 视力异常、呼吸异常、出血、疑似感染、持续加重疼痛
- 最低价、底价、必须比别人低、强压价

### 路由规则

- 医疗/安全/投诉类风险优先级高于所有业务意图。
- 当前消息优先级高于历史对话。
- 如果主路由已输出 `need_human=true`，后续节点不得改写为普通回复。

## P0-3 禁止外部数据幻觉

### 适用流程

- `SF6_store_match`
- `SF7_price_consult`
- `SF8_campaign_push`
- `SF9_appointment`
- `DIRECT_REPLY`

### 立即规则

没有对应外部数据时，禁止输出：

- 门店名称、地址、营业时间、停车、地铁、地图链接
- 活动名称、活动权益、活动价、有效期、适用门店
- 具体价格、定金、优惠、学生价、最低价
- 已预约、预约成功、已取消、已改约、某时段可以
- 门店数量、医生排班、皮肤检测是否支持

### 统一兜底

门店缺失：

```text
我先帮你匹配近一点的门店。你方便说下所在城市或常去区域吗？
```

价格缺失：

```text
这个项目当前还缺少可直接确认的价格信息，我先帮你确认具体项目/规格后再给你准确价格。
```

活动缺失：

```text
活动要看具体项目和当前有效期，我先帮你确认。你想了解哪个项目方向？
```

预约缺失：

```text
我先帮你确认这个时间段是否还有位置，确认好再回复你。
```

## P0-4 全局禁止内部词和占位符

### 禁止客户侧输出

- 系统价
- 系统配置
- 系统里没有
- 查询系统
- 数据库
- 价格库
- 知识库
- 检索结果
- 工具返回
- `items_outputList`
- `store_info`
- `campaign_info`
- `[请根据...]`
- `[活动结束日期]`
- `[适用门店]`
- `XX店`
- `XX市XX区XX路XX号`
- `X号线XX站`

### 立即做法

在每个子流程提示词末尾增加强约束：

```text
最终回复前自检：如果 content 中包含内部变量名、系统/知识库/检索/工具字样、模板占位符、XX地址、XX门店，必须改写为客户可理解的自然表达；无法确认的信息不要输出。
```

## P1-1 修复 DIRECT_REPLY 短确认和到店前问题

### 短确认规则

当前消息为“好的、嗯嗯、收到、可以、谢谢、先这样、晚点再说”时，先看上一轮助手动作：

- 上文是转人工确认，进入 `HUMAN_HANDOFF`
- 上文是预约确认，进入 `SF9_appointment`
- 上文是普通说明，进入 `DIRECT_REPLY`
- 上文是价格说明，不要重新问价格

### 到店前问题模板

带什么：

```text
正常带手机和身份证就可以，尽量素颜或少化妆，方便现场评估。
```

做多久：

```text
具体要看项目和现场评估，建议预留充足一点时间，以门店安排为准。
```

能不能化妆：

```text
建议尽量素颜或少化妆过去，现场评估会更方便。
```

是否空腹：

```text
一般皮肤类面诊不需要特意空腹，具体以到店项目安排为准。
```

## P1-2 修复 SF3/SF4 过早推荐

### 立即规则

客户只是描述痛点时，不直接列项目名，先问 1 个关键区分问题。

例如：

```text
你这个方向我先帮你判断一下。你更主要是毛孔粗大出油，还是肤色暗沉更明显？
```

客户明确问“有什么项目推荐”时，才能给 1-2 个方向，并加面诊边界。

## P1-3 修复 SF5 竞品和砍价边界

### 竞品强识别词

包含以下词时优先进入 `SF5_competitor_response`：

- 别家
- 另一家
- 他们家
- 其他机构
- 同行
- 报价单
- 截图
- 比你们便宜
- 他们说
- 朋友在别家做

### 分流

- “别家便宜/为什么贵/报价差异”：`SF5`
- “最低价/底价/必须便宜/不便宜就不来”：`HUMAN_HANDOFF` 或强价格边界

## P1-4 修复 SF10 图片和承诺感

### 立即规则

- 每轮最多 1 张图片。
- 图片必须和顾虑强相关。
- 禁止“确保正品、不会隐形消费、可以放心、保障安全、一定有效”等承诺式表达。

### 安全替代表达

- “到店可以核对产品来源”
- “费用会提前确认”
- “操作前会评估并说明风险”
- “不适合的情况不建议做”

## P1-5 修复 SF12 投诉退款和非本机构售后

### 投诉退款

命中退款、投诉、效果纠纷、强烈不满时，必须转人工，不得继续售后护理科普。

### 非本机构售后

```text
可以先帮你梳理风险点，但具体处理建议联系原操作机构或专业医生确认；如果不适明显加重，建议及时就医。
```

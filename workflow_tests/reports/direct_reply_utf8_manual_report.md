# DIRECT_REPLY UTF-8 测试人工结论

测试时间：2026-05-22

## 结论

DIRECT_REPLY 已完成 24 条 UTF-8 测试。接口错误 0 条，空回复 0 条。

当前 DIRECT_REPLY 边界不稳定。短确认类大多能短回复，但到店前问题大量兜底成欢迎语；地址/停车仍然编造；“收到”被误判为价格咨询；客户拒绝/暂停时有时仍继续提预约；同意转人工没有转人工。

## 表现较好的点

- “谢谢、没事了、再看看、我先问问朋友、我在路上”等轻量消息回复较短。
- “算了/不用了/晚点再说”多数没有继续强推项目。
- 改约、取消等至少能识别成对应意图，没有完全当闲聊。

## 主要问题

1. 到店前问题兜底欢迎语  
   `DR_007` “明天去要带什么”  
   `DR_008` “到店要做多久”  
   `DR_009` “可以化妆过去吗”  
   `DR_010` “需要空腹吗”  
   都输出“您好，我是您的专属客服...”。  
   这些应是 DIRECT_REPLY 的典型场景，必须有通用短答。

2. 地址/停车继续编造  
   `DR_013` 编造厦门思明店停车、地铁镇海路站。  
   `DR_014` 输出 `XX市XX区XX路XX号`、停车券、营业时间。  
   这和 SF6 同类问题一致：地址/交通/停车必须只来自门店库。

3. 短确认上下文理解失败  
   `DR_018` “收到” 被判成 `price_inquiry`，继续问价格。  
   当前消息是短确认时，应优先按上一轮助手内容做轻确认，而不是从历史里抽价格意图。

4. 拒绝/暂停被错误处理为取消预约成功  
   `DR_016` “我不考虑了” 回复“帮你把预约记录取消了”。  
   客户没明确说取消预约，不能直接取消。

5. 停止联系请求兜底失败  
   `DR_023` “先别联系我了” 回复欢迎语。  
   应短回复“好的，我先不打扰你”，并停止营销触达。

6. 同意转人工没有转人工  
   `DR_024` 上文是“我帮你安排真人顾问接上，可以吗？”，客户说“可以”，结果回复欢迎语。  
   这种必须进入 HUMAN_HANDOFF。

7. 风格小问题  
   多条使用 emoji 或过于轻松的“好哒～”，企微医美客服建议更稳一些。

## 必须调整的规则

1. 短确认优先规则  
   当前消息为“好的、嗯嗯、收到、可以、谢谢、先这样、晚点再说”时，必须结合上一轮助手动作判断：
   - 上文是普通说明：DIRECT_REPLY
   - 上文是转人工确认：HUMAN_HANDOFF
   - 上文是预约确认：SF9
   - 上文是价格说明：DIRECT_REPLY，不要重新问价格

2. 到店前问题补 DIRECT_REPLY 模板  
   需要覆盖：
   - 带什么
   - 做多久
   - 能不能化妆
   - 是否空腹
   - 到了/在路上

3. 停止触达规则  
   “别回了、不用回复、先别联系我、不考虑了”：
   - 不追问
   - 不推荐
   - 不预约
   - 不取消已有预约，除非客户明确说“取消预约”

4. 地址/停车/路线不属于 DIRECT_REPLY  
   统一走 SF6，且必须用门店库。没有门店信息不输出地址。

5. 预约变更不直接完成  
   改约/取消应走 SF9，并依赖预约接口。

## 建议模板

带什么：
“明天正常带身份证/手机就可以，尽量素颜或少化妆，方便现场评估。”

做多久：
“一般要看具体项目和现场评估，建议预留充足一点时间，具体以门店安排为准。”

晚点再说：
“好的，那我先不打扰你，后面想继续了解再找我就行。”

先别联系：
“好的，我先不打扰你。”

同意转人工：
“可以，我帮你安排顾问接上。”

## 基础信息

**基础路径**: `/platform_agent`

**环境**:

*   `正式环境`: `https://www.henm.cn\` 已上线
    
*   `测试环境`: `https://v2.henm.cn\`  已上线
    

**认证方式**:

*   认证通过固定 API Token 校验，Token 由后端配置
    
*   Header 名称: `token` 传: `8d3f2e7a-1b5c-4a6f-9e8d-7c2b3a1f5e6d`
    
*   Header 名称: `Request-From` 传: `platform_agent`
    
*   所有接口请求参数中需携带 `user_id`（员工ID）暂时不用
    
*   所有接口请求参数中需携带 `corp_Id`（企微号）
    
*   所有接口请求参数中需携带 `wechat`（微信号）
    

**免认证接口**:

**统一返回格式**:

成功响应:

```json
{
  "code": 200,
  "msg": "success",

  "data": [ ]

}

```

失败响应:

```json
{
  "code": 400,
  "msg": "错误信息",

  "data": [ ]

}

```

**常见错误码**:

*   401: 未登录或Token失效
    
*   403: 无权限访问
    
*   404: 路由不存在
    
*   500: 服务器内部错误

## 二、社区模块

### 2.1 获取筛选项

**接口地址**: `GET /platform_agent/option`

**接口说明**: 按需获取各类筛选项数据（门店、项目、用户、预约金等）

**请求参数**:

| 参数名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| option | string | 否 | 筛选项类型，多个用竖线分隔，如: store/category/user,不传则返回默认选项（storecategory、user、prepay、priceLimit） |
| store\_id | int | 否 | 门店ID，当 option 包含 `categoryFiltration` 时必填 |

**可用的筛选项类型**:

#### 1. `store` - 门店列表

返回所有门店信息

**返回字段**:

*   `id` (int) - 门店ID
    
*   `department_id` (int) - 部门ID
    
*   `name` (string) - 门店名称
    
*   `initial` (string) - 首字母
    
*   `zone_id` (int) - 区域ID
    
*   `business_hours_begin` (string) - 营业开始时间
    
*   `business_hours_end` (string) - 营业结束时间
    
*   `status` (int) - 状态
    
*   `shore_show` (int) - 是否展示
    
*   `schedule_status` (int) - 排班状态
    
*   `plan_status` (int) - 计划状态
    
*   `region_id` (int) - 区域ID
    
*   `is_pause` (int) - 是否暂停
    
*   `pause_start` (string|null) - 暂停开始时间
    
*   `pause_end` (string|null) - 暂停结束时间
    

#### 2. `category` - 项目树

返回项目分类树形结构

**返回字段**:

*   `id` (int) - 项目ID
    
*   `name` (string) - 项目名称
    
*   `children` (array) - 子项目列表
    

#### 3. `user` - 用户列表

返回所有工作人员信息

**返回字段**:

*   `id` (int) - 用户ID
    
*   `name` (string) - 用户姓名
    
*   `department_id` (int) - 部门ID
    

#### 4. `prepay` - 预约金选项

返回预约金配置列表

**返回字段**: 根据实际配置返回

#### 5. `categoryFiltration` - 过滤项目

根据门店ID过滤可用项目（需传 store\_id 参数）

**额外参数**: `store_id` (必填)

**返回字段**:

*   `id` (int) - 项目ID
    
*   `name` (string) - 项目名称
    

**说明**: 返回在预约金上架且该门店有库存的项目

#### 6. `priceLimit` - 折扣限制

**返回字段**:

*   `min_discount_ratio` (int) - 最小折扣比例 会返回60,就是6折
    

**请求头**:

*   `token`: Token (必填)
    

**请求示例**:

```plaintext
# 获取默认选项
GET /platform_agent/community/option/index

# 获取指定选项
GET /platform_agent/community/option/index?option=store|category|user

# 获取过滤项目（需要门店ID）
GET /platform_agent/community/option/index?option=categoryFiltration&store_id=10

```

**返回示例**:

## 四、排班模块 (Schedule)

### 基本信息

| 项目 | 说明 |
| --- | --- |
| 请求路径 | `GET /order/schedule/hours_list` |
| 请求方式 | `GET` |
| 用途 | 获取指定门店在指定日期的排客时间表，包含座位分配、订单占用、门店信息及池化状态 |

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `store_id` | `int` | 是 | 门店ID |
| `date` | `string` | 是 | 日期，格式 `YYYY-MM-DD` |

### 响应结构

```json
{
  "code": 200,
  "msg": "ok",
  "data": {

    "hours": [ ],

    "extend": {},
    "enable_pool": {},

    "bed_types": [ ]

  }
}

```

#### `data.hours` — 时间段列表 `HourSlot[ ]`

每个时间段按门店配置的间隔时间划分（默认 3600 秒 = 1 小时，可通过门店日常配置 `service_rules.interval_time` 自定义）。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `time` | `string` | 时间段标识。正常时段为 `"HH:mm"` 格式（如 `"09:00"`）；额外新增时段显示为 `"新增"` |
| `start` | `int` | 时间段开始的 Unix 时间戳（额外新增时段为 `0`） |
| `end` | `int` | 时间段结束的 Unix 时间戳（额外新增时段为 `0`） |
| `new_qty` | `int` | 新客座位数量 |

| `new` | `Seat[ ]` | 新客座位列表 |

| `old_qty` | `int` | 老客座位数量 |

| `old` | `Seat[ ]` | 老客座位列表 |

> 当额外新增时段（`time="新增"`）的 `new` 和 `old` 数组均为空时，该时间段不会返回。

#### `Seat` — 座位对象

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `time` | `string` | 座位时间标识，`"HH:mm"` 格式；额外新增床位显示为 `"新"`；被订单占用后会修改为订单的实际预约时间 |
| `start` | `int` | 座位对应的开始时间戳（额外新增为 `0`） |
| `end` | `int` | 座位对应的结束时间戳（额外新增为 `0`） |
| `is_insertion` | `bool` | 是否为加塞床位（插入到已有时间段内的额外床位） |
| `is_released` | `bool` | 是否为释放的专属池座位 |
| `bed_type_id` | `int` | 床位类型ID（仅池化模式下存在）。`0` = 公共池，`>0` = 专属池 |
| `order` | `object\|null` | 占用该座位的订单信息，无占用时为 `null` |
| `addon_seat_id` | `int` | 额外座位配置ID（仅插入类型床位存在） |
| `start_time` | `string` | 插入床位的开始时间 `"HH:mm"`（仅插入类型床位存在） |

#### `Seat.order` — 订单对象（座位被占用时）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `int` | 订单ID |
| `plan_at` | `int` | 排客（预约到店）时间戳 |
| `customer_kind` | `int` | 客户类型：`1` = 新客，`2` = 老客 |
| `schedule_kind` | `int\|bool` | 排客日志类型：`12` = 排客，`15` = 到店，`13` = 取消排客，`31` = 改约 |

| `origin_plan_at` | `int` | 原始排客时间（仅改约时存在，当 plan\_at 与日志中记录不同时出现） |
| `plans` | `array` | 订单项目列表 |
| `consumes` | `array` | 订单消耗品列表 |

| 字段 | 类型 | 说明 |
| --- | --- | --- |

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `new` | `bool` | 新客是否启用池化算法分配 |
| `old` | `bool` | 老客是否启用池化算法分配 |

#### `data.bed_types` — 床位类型列表（仅池化模式下非空）

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `int` | 床位类型ID |
| `name` | `string` | 床位类型名称 |

| `allocation_names` | `string[ ]` | 关联品项分类名称数组 |

### 业务逻辑说明

1.  **时间间隔：** 每个时间段的长度由门店日常配置 `service_rules.interval_time` 决定，默认 3600 秒。
    
2.  **座位数量计算优先级：** 值班老师技能计算 → 门店日常配置覆盖 → 额外床位叠加。
    
3.  **池化算法：** 启用后总床位按比例（`RATIO`）或数量（`COUNT`）分配到各专属池和公共池，使用最大余数法分配。
    
4.  **释放规则：** 专属池空闲座位满足释放条件后标记 `is_released = true`，支持 `daily_time`（指定时间点释放）和 `before_expire`（时间段开始前 N 小时释放）两种触发方式。
    
5.  **历史 vs 未来日期：** 过去日期通过排客日志还原当时的座位占用情况；未来日期直接查询当前有效订单。

## 2. 获取可预约时间 — `available_time`

### 基本信息

| 项目 | 说明 |
| --- | --- |
| 请求路径 | `GET /order/schedule/available_time` |
| 请求方式 | `GET` |
| 用途 | 获取指定门店在指定日期的可预约时间点列表，用于创建预约时选择可用时间 |

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `store_id` | `int` | 是 | 门店ID |
| `date` | `string` | 是 | 日期，格式 `YYYY-MM-DD` |

### 响应结构

```json
{
  "code": 200,
  "msg": "ok",
  "data": {
    "new": ["09:00", "10:00", "11:00"],
    "new_addon": ["09:30"],
    "old": ["09:00", "10:00"],
    "old_addon": ["10:15"],
    "pre": ["09:00", "09:30", "10:00", "10:30"]
  }
}

```

#### `data` 字段说明

| 字段 | 类型 | 说明 |
| --- | --- | --- |

| `new` | `string[ ]` | 新客可预约的正常时间点列表（`"HH:mm"` 格式），仅包含当前时间之后的可用时间 |

| `new_addon` | `string[ ]` | 新客可预约的加塞时间点列表（`"HH:mm"` 格式） |

| `old` | `string[ ]` | 老客可预约的正常时间点列表（`"HH:mm"` 格式），仅包含当前时间之后的可用时间 |

| `old_addon` | `string[ ]` | 老客可预约的加塞时间点列表（`"HH:mm"` 格式） |

| `pre` | `string[ ]` | 可用于预排（提前占位）的时间点列表（`"HH:mm"` 格式），不过滤已过期时间 |

### 业务逻辑说明

1.  **可用时间判定：** 座位未被订单占用 且 座位结束时间晚于当前时间。
    
2.  **池化模式下的处理：**
    
    *   **池化开启时：** 可用时间不再依赖座位表，而是按 15 分钟间隔（900 秒）从每个时间段的 `start` 到 `end` 生成所有时间点（仅保留未过期的）。加塞时间仅从座位中提取 `is_insertion = true` 且未被占用的时间。
        
    *   **非池化模式：** 直接从座位表筛选未占用座位，按 `is_insertion` 区分正常时间和加塞时间。
        
3.  **预排时间（pre）计算：** 不区分新老客和池化模式，遍历每个正常时间段：
    
    *   如果时间段的起始时间等于 `hour.start` 且持续时长 >= 半个间隔时间，加入该起始时间。
        
    *   如果时间段的结束时间等于 `hour.end` 且持续时长 >= 半个间隔时间，加入起始时间 + 半个间隔时间。
        
    *   预排时间不过滤已过期时间。
        
4.  **数据来源：** 内部调用 `getHoursList()` 方法，与 `/hours_list` 使用相同的底层逻辑，只是对结果做二次提取和简化。

## 七、支付模块 (Pay)

### 7.1 预支付

**接口地址**: `POST /platform_agent/pay/prepay`

**接口说明**: 创建预支付订单

**请求参数**:

| 参数名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| customer\_id | int | 是 | 客户ID |
| order\_id | int | 是 | 订单ID |
| payment\_id | int | 是 | 付款方式ID 传 12 |
| collection\_id | int | 是 | 收款记录ID |

**请求示例**:

```json
{
  "customer_id": 100,
  "order_id": 12345,
  "payment_id": 1,
  "collection_id": 500
}

```

**请求头**:

*   `token`: Token (必填)
    

**返回参数**:

| 参数名 | 类型 | 说明 |
| --- | --- | --- |
| status | int | 1:未支付;2:已支付;3:支付中;4:已作废 |

**返回示例**:

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "status": 2
  }
}

```

**错误说明**:

*   用户手机号码不能为空
    
*   客户必填
    
*   订单不存在
    
*   付款方式不存在，或者未选择收款记录
    
*   付款记录不存在

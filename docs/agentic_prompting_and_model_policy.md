# Agentic 提示词结构与模型选择规则

## 1. 目标

本规则用于指导 AI Paths 客服系统后续的 LangGraph 节点、业务 skill、工具封装和模型选择。

系统目标不是把 Coze 里的 `SF1-SF12` 原样搬到代码里，而是形成一个受控的 Agentic 客服大脑：

- 当前消息可以同时触发多个业务能力。
- 简单任务用规则或快模型，复杂任务再用强模型。
- 所有外部事实必须来自工具、知识库、接口或客户明确表达。
- 每个节点有明确输入、输出、边界和日志。
- 高风险、投诉、退款、未成年、严重术后异常等场景由硬规则兜底。
- 流程尽量短、稳定、可降级，不能为了“智能”堆叠过多模型节点。

## 1.1 客服人格与表达边界

统一客服身份：

- 客服对外可以自称“小美”。
- 不主动透露自己是 AI，也不使用“AI客服”“机器人”“系统”等生硬身份描述。
- 如果客户直接问“你是 AI 吗”，可以自然回应为“小美先帮你把问题处理清楚，涉及具体判断我也会同步专业同事确认”，不要陷入身份争辩。

转人工/升级表达：

- 内部可以仍然使用 `HUMAN_HANDOFF`。
- 对客户不默认说“转人工”“转接人工”。
- 优先表达为：
  - “这个我让专业人士协助你看下。”
  - “这个情况我帮你同步给门店同事确认下。”
  - “涉及具体处理，我让专业同事接着协助你。”
  - “你这个情况需要护理老师/专业同事看会更稳妥。”

禁止表达：

- “我是 AI 客服”
- “系统判断需要转人工”
- “工具返回”
- “知识库显示”
- “根据流程”
- “机器人无法处理”

除非客户明确要求解释系统机制，否则回复只呈现服务结果，不暴露内部流程。

## 2. 从 Codex / Claude Code 类工具借鉴的结构

### 2.1 项目级规则文件

Codex 的 `AGENTS.md`、Claude Code 的 `CLAUDE.md` 都体现同一个原则：长期有效的项目规则不应该散落在对话里，而应该写进仓库文件。

本项目采用：

- `AGENTS.md`：当前项目最高层操作规则。
- `docs/agentic_customer_service_design.md`：客服 Agentic Workflow 总体设计。
- `docs/langgraph_coze_tool_contracts.md`：Coze 工具工作流契约。
- `docs/agentic_prompting_and_model_policy.md`：提示词结构、skill 结构和模型路由规则。

后续新增模块时，先查这些文档，再写代码或提示词。

### 2.2 聚焦的子智能体 / skill

Claude Code subagent 的核心价值不是“多开几个大模型”，而是让每个子能力只有一个清晰职责、自己的上下文和有限工具权限。

本项目里的业务 skill 应该这样设计：

- `project_consult_skill`：只负责项目咨询事实和回复要点。
- `price_consult_skill`：只负责价格事实、预算解释和报价边界。
- `trust_build_skill`：只负责正规性、资质、产品真假、售后保障等信任问题。
- `competitor_skill`：只负责竞品对比，不跟价、不诋毁。
- `after_sales_skill`：只负责售后护理和风险升级。
- `store_skill`：只负责门店、地址、导航、停车。
- `appointment_skill`：只负责预约查询、改约、取消、可约时间。

skill 不直接垄断最终回复。它输出结构化事实和 `reply_points`，最终由 `reply_synthesizer` 统一合成。

### 2.3 工具权限最小化

Claude Code subagent 支持限制工具权限；OpenAI Agents SDK 强调应用自己掌控工具执行、状态和审批。

本项目规则：

- 主脑可以规划工具，但不能直接编造工具结果。
- 每个 skill 只能调用自己需要的工具。
- 价格 skill 可以查价格库和价格知识库，不能查门店接口。
- 门店 skill 可以查门店匹配和门店详情，不能生成项目报价。
- 售后 skill 可以查订单和售后知识库，严重风险必须交给硬规则或人工。
- `reply_synthesizer` 默认不直接查工具，只消费上游结果。

### 2.4 Hook / Guardrail 思维

Claude Code hooks 的价值是：在模型执行前后增加确定性检查。

本项目必须保留硬规则节点：

- `normalize_input`：清洗输入、阻断乱码 query。
- `hard_guardrails`：投诉、退款、未成年、孕妇、严重术后异常、人工诉求。
- `tool_result_validator`：检查工具返回是否为空、是否乱码、是否缺关键字段。
- `reply_validator`：检查回复是否编造价格、承诺效果、泄露工具过程、输出格式错误。

这些不是提示词建议，而是代码级或 schema 级约束。

### 2.5 全链路 tracing

OpenAI Agents SDK 的 tracing 思路适合本项目：每次运行要记录模型调用、工具调用、handoff、guardrail 和自定义事件。

本项目每轮必须记录：

- request_id
- 节点名
- 输入快照
- 输出快照
- 模型名称
- token / 耗时，如果可得
- 工具调用参数摘要
- 工具返回摘要
- 错误和降级路径

日志目标：让问题能定位到“主脑规划错了、工具查错了、知识库没命中、回复合成生硬、还是前端传参错了”。

## 3. 标准提示词结构

后续每个模型节点提示词都必须包含以下结构，不能只写一段角色描述。

```text
角色：
你是什么节点，不是什么节点。

目标：
本轮必须完成什么。

输入：
列出可用输入字段，以及字段优先级。

可用工具/资料：
列出允许使用的工具、知识库、上游结果。

决策规则：
按优先级列出必须遵守的判断顺序。

允许行为：
说明可以推断、可以合并、可以追问的范围。

禁止行为：
说明不能编造、不能承诺、不能转移话题、不能泄露系统过程。

输出格式：
给出严格 JSON schema。

失败/降级：
工具为空、信息不足、图片不清晰、上下文冲突时怎么输出。
```

## 4. Skill 输出结构

所有业务 skill 默认输出结构：

```json
{
  "skill": "",
  "intent": "",
  "facts": [],
  "reply_points": [],
  "missing_slots": [],
  "risk_flags": [],
  "suggested_next_step": "",
  "confidence": 0
}
```

字段要求：

- `facts` 只能放来自工具、知识库、接口、客户原话或图像理解的事实。
- `reply_points` 是给最终回复合成器用的要点，不是完整客服话术。
- `missing_slots` 只放必要缺口，不要为了流程完整强行追问。
- `risk_flags` 用于标记高风险、人工、医疗、投诉、价格纠纷等。
- `suggested_next_step` 必须克制，优先推动项目理解或到店面诊，但不能强推预约。

## 5. 主脑 Planner 规则

`planner_brain` 不是传统单路由分类器，而是动作规划器。

输入：

- 当前消息
- 近几轮对话
- 画像和事件
- 当前预约/订单/门店缓存
- 图片理解结果
- 硬规则结果

输出：

```json
{
  "primary_goal": "",
  "detected_intents": [],
  "actions": [
    {
      "type": "skill_or_tool",
      "name": "",
      "reason": "",
      "priority": 1
    }
  ],
  "reply_strategy": {
    "max_messages": 3,
    "must_answer": [],
    "may_guide_to": "",
    "must_not": []
  },
  "confidence": 0
}
```

规划限制：

- 每轮最多 3 个主要业务意图。
- 每轮最多 4 个工具/skill 动作。
- 高风险命中后不再规划普通营销回复。
- 工具可以并行，skill 输出必须统一交给回复合成。
- 不确定时优先轻量澄清，不要为了走流程硬查接口。

## 6. 复杂度分级

核心原则：能不用模型就不用模型；能用快模型就不用强模型；能一步完成就不要拆成多轮链式推理。

### L0：确定性任务，不用模型

适用：

- JSON 清洗、字段映射、时间格式化。
- Coze workflow 返回解析。
- 价格 DB 精确查询。
- 明确关键词硬规则，例如投诉、退款、未成年、人工。
- 输出 schema 校验。

策略：

- 用 Python 代码完成。
- 不调用任何模型。
- 优先级最高，能覆盖的场景不要再交给模型。

### L1：简单理解任务，用最快模型

适用：

- 单意图分类。
- query 改写。
- 简单槽位提取。
- 简短画像增量。
- 普通知识库检索词生成。

模型建议：

- 优先使用速度最快、成本最低的文本模型。
- 可选：Qwen Flash / Turbo 类，Doubao Lite / Mini 类。

策略：

- 输出严格 JSON。
- 温度低。
- 不承担最终客服语气。
- 不做长链推理，不调用多工具。

### L2：中等复杂任务，用均衡模型

适用：

- 多意图规划。
- 多个工具结果合并前的结构化总结。
- 售后、竞品、信任问题的普通级别回复要点。
- 画像与事件抽取，且输入包含多轮历史或图片描述。

模型建议：

- Qwen Plus 类。
- Doubao Pro / Seed Lite-Pro 类。

策略：

- 可处理上下文和轻度冲突。
- 仍要求结构化输出。
- 对医疗、投诉、退款不自由发挥。
- 如果可以通过一个业务 skill 解决，不再拆多个模型节点。

### L3：高复杂 / 高敏感任务，用强模型

适用：

- 多意图最终回复合成。
- 同时包含图片面诊、价格、信任、预约推进。
- 投诉边缘、强烈不信任、竞品压价、售后焦虑等敏感语气。
- 工具结果互相冲突，需要克制解释。

模型建议：

- Qwen Max 类。
- Doubao Seed Pro / Thinking 类。

策略：

- 只在确实需要时使用。
- 必须消费上游事实，不能自行补事实。
- 输出后走 `reply_validator`。
- 高复杂模型调用要少而关键，避免每轮默认使用。

### LV：视觉理解任务，用视觉模型

适用：

- 面部皮肤图、眼周图、身体局部图。
- 竞品报价截图、聊天截图、活动海报、门店地图。
- 付款凭证、产品包装、报告/病例。

模型建议：

- Qwen VL Plus / VL Max 类。
- Doubao Vision / 多模态模型。

策略：

- 输出 `image_info`，不直接回复客户。
- 只描述可见事实，不诊断。
- 高风险图片直接标记风险并建议人工。

## 7. 各模块默认模型策略

| 模块 | 默认复杂度 | 默认策略 |
|---|---:|---|
| `normalize_input` | L0 | 纯代码 |
| `hard_guardrails` | L0 + L1 | 关键词和规则优先，模糊风险再用快模型 |
| `image_understanding` | LV | 视觉模型，输出结构化 `image_info` |
| `planner_brain` | L1/L2 | 单意图用快模型，多意图或上下文冲突用均衡模型 |
| `project_consult_skill` | L1/L2 | query 改写用快模型，复杂项目解释用均衡模型 |
| `price_consult_skill` | L0/L1 | 价格事实由 DB，query/槽位用快模型 |
| `trust_build_skill` | L2/L3 | 普通信任用均衡模型，强不信任或需要图片资料时用强模型 |
| `competitor_skill` | L2/L3 | 普通比价用均衡模型，压价/指控/截图复杂时用强模型 |
| `after_sales_skill` | L0/L2 | 风险规则优先，普通护理用均衡模型 |
| `store_skill` | L0/L1 | 门店事实接口优先，文本整理用快模型或模板 |
| `appointment_skill` | L0/L1 | 预约事实接口优先，缺槽澄清用快模型 |
| `reply_synthesizer` | L2/L3 | 单意图简单回复用均衡模型，多意图/敏感回复用强模型 |
| `profile_event_extractor` | L1/L2 | 短消息用快模型，多轮和图片上下文用均衡模型 |

## 8. 模型升级条件

任一条件命中时，可以从 L1 升级到 L2，或从 L2 升级到 L3：

- 当前消息包含 2 个以上业务意图。
- 同时存在图片理解结果和文本诉求。
- 客户表达强不信任、焦虑、愤怒、质疑正规性。
- 客户提到竞品价格、截图、同价、底价。
- 客户已有预约，但又在咨询项目/价格/地址，需要跨流程提醒。
- 工具结果为空，但仍需要给出克制可用回复。
- 历史画像与当前消息存在冲突。
- 回复会影响到店转化或投诉风险。

## 9. 模型降级条件

任一条件命中时，应使用更快模型或纯代码：

- 只是字段解析、格式转换、排序、去重。
- 只是明确单意图，例如“水光多少钱”“地址发我”。
- 工具结果已经足够，模型只需轻量改写。
- 客户只说“好、收到、谢谢、我看看”。
- 输出是固定 JSON 或模板。
- 当前节点只是把上游 facts 改写成 1 条普通客服话术。

## 9.1 稳定性优先规则

后续设计节点时遵守：

- 单轮默认链路不要超过：输入标准化 -> 硬规则 -> 规划/轻分类 -> 必要工具 -> 回复合成 -> 画像事件。
- 不要让多个大模型串行判断同一件事。
- 不要让回复合成器重新做工具选择。
- 不要让工具节点生成最终话术。
- 工具失败时必须有降级回复，不允许整轮空回复。
- 多意图最多处理 3 个，超过时先回答最重要的并轻量承接剩余问题。
- 高风险场景宁愿短回复升级专业人士，也不要长篇解释。

## 10. 回复合成取舍

最终回复必须先解决客户显性问题，再轻度推进下一阶段。

排序原则：

1. 风险和人工。
2. 信任顾虑。
3. 当前问题事实。
4. 项目/价格解释。
5. 门店/预约推进。

推进规则：

- 可以引导客户继续补充照片、斑型、部位、预算、城市。
- 可以建议“更准确需要结合照片/到店面诊看”。
- 可以提醒已有预约。
- 不要每轮都强行预约。
- 不要在客户明显抗拒时继续推进。
- 需要自称时用“小美”，不要过度亲昵。
- 不使用“亲亲”“宝宝”“系统查询到”“知识库显示”等表达。

### 10.1 不踢皮球规则

客服回复以解决客户当前问题为主，不能把所有问题都推回“需要门店确认”“需要看具体情况”。

必须优先做到：

- 如果工具、知识库、客户画像、图片理解或客户原话里已有可用事实，先给答案或判断方向。
- 如果没有价格数字，但已经知道项目名，要承接项目名，例如“皮秒价格要看具体配置”，不要再问“您想问哪个项目”。
- 如果客户问适合什么项目，先给 1-2 个方向和判断依据，再问必要补充信息。
- 如果客户问信任/正规，先解释可核验维度或可发送资料，不要只说“需要专业人士确认”。
- 如果客户问地址/停车，已有门店信息时直接给地址/停车信息，不要重复问城市。
- 如果客户问售后轻微护理，先收集必要信息，但不能直接说“我不知道/等人工”。

允许谨慎表达，但要短：

- “具体要结合斑的深浅和范围”
- “最终以当前项目配置为准”
- “严重不适我让专业同事协助看”

禁止高频使用：

- “这个需要具体情况”
- “这个要到店才知道”
- “需要门店确认”
- “建议咨询人工”
- “我无法判断”

这些表达只能作为补充边界，不能成为整条回复的主体。

### 10.2 每次测试后的回复质量自检

每次实现或修改后，测试报告必须包含“回复质量判断”：

- 是否回答了客户显性问题。
- 是否利用了已知事实和历史画像。
- 是否有不必要的推诿或重复追问。
- 是否保留合规边界但不过度免责声明。
- 是否符合“小美”客服口吻。
- 是否需要继续优化。

## 11. 后续实现验收标准

每新增一个模块，必须满足：

- 有输入/输出 schema。
- 有复杂度等级和默认模型。
- 有工具权限边界。
- 有失败降级路径。
- 有日志记录。
- 有至少 3 个测试样例：
  - 简单正常输入
  - 多意图输入
  - 工具为空或风险输入

如果一个模块无法说明这些内容，先不要实现。
